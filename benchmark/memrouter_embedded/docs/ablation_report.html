3. 重点分析：IntentAnalyzer Token 节省有限

3.1 Token 构成拆解

以 Baseline 为例，总 token 消耗构成如下：

Baseline

3.2 为什么节省有限？

因素说明
Bot token 占绝对主导 每次回答生成消耗 ~30K tokens（prompt 31K + completion 0.7K），81 题共 ~2.6M tokens
IntentAnalyzer 占比小 Baseline 中 IntentAnalyzer 仅占 ~4.5%（~123K / 2.7M），且单次仅 ~1,500 tokens
MemRouter 引入新开销 Treatment 用 LLM Fallback（418 tokens/次）替代 IntentAnalyzer，但 Fallback 次数（22次）与 IntentAnalyzer 次数（81次）不同
Template Hit 已零 token 59/81 题（72.8%）通过模板匹配零 token 路由，这部分已经是最优状态

核心结论：MemRouter 确实消除了 IntentAnalyzer 的 token 消耗（81次 × 1,500 tokens），并通过模板匹配实现了 59/81 题的零 token 路由。但由于 Bot 回答生成是 token 消耗的绝对大头（95%+），这一节省在总成本中仅占约 4%，反映在 DeepSeek 平台上仅为几分钱的差距。

关于准确率：Baseline 两次运行相差 12.35pp，说明 81 题的样本量下随机方差较大。若以 Baseline #1（69.14%）为基准，Treatment v2（62.96%）甚至略低；但 Baseline #2（56.79%）显示 Treatment 有明显提升。建议在更大样本或多次运行后取平均，以降低方差影响。

3.4 若需显著降低总成本

若要显著降低实验总成本，需要优化的是 Bot 回答生成阶段的 prompt 长度，而非 Routing 阶段。可能的方向：

  • 减少检索到的记忆片段数量（当前可能传入过多记忆到 bot prompt）
  • 使用更轻量的模型进行回答生成（如从 deepseek-v4-flash 降级到更小模型）
  • 对记忆内容进行压缩/摘要后再传入 bot
